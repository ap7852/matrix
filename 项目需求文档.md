## 第一章 产品概述

### 1.1 产品背景

Element X HarmonyOS 是面向鸿蒙原生生态的 Matrix 协议即时通讯客户端，以 Element X Android 为 1:1 复刻标的，旨在填补鸿蒙生态中缺乏高质量、端到端加密即时通讯应用的空白。截至 2026 年 3 月，华为 AppGallery 已拥有超过 35 万个原生鸿蒙应用，但尚无任何 Matrix 协议客户端，本项目具有显著的生态先发优势。

### 1.2 产品定位

基于 Matrix 2.0 协议栈（Native Sliding Sync + OAuth/OIDC + Element Call）的鸿蒙原生即时通讯客户端，提供与 Element X Android 功能对等的安全、私密、去中心化通讯体验。

### 1.3 目标用户

- 已使用 Element X Android/iOS 的 Matrix 用户，需要在鸿蒙设备上延续使用体验
- 注重隐私与数据主权的个人用户
- 需要私有化部署通讯方案的企业用户（Synapse/Conduit 自建服务器）

### 1.4 版本策略

HarmonyOS API 22（HarmonyOS 6.0.2）已于 2026 年 1 月正式发布，**本项目直接以 API 22 为开发与发布基准**，最低支持版本设定为 API 22，不再需要向下兼容 API 21 的过渡策略。

## 第二章 功能需求

功能优先级采用 MoSCoW 分级：**P0（Must Have）** / **P1（Should Have）** / **P2（Could Have）** / **P3（Won't Have for now）**。

### 2.1 认证模块（Authentication）

#### 2.1.1 服务器配置

**P0** 用户可通过输入 Homeserver 地址（如 `matrix.org`）或扫描二维码完成服务器配置。客户端需通过 `/.well-known/matrix/client` 自动发现服务器能力，区分是否支持原生 Sliding Sync（MSC4186）和 OAuth/OIDC（MSC3861）。

**P0** 服务器地址输入框需实时校验格式合法性，并给出明确的错误提示（如"无法连接到该服务器"、"该服务器不支持 Matrix 协议"）。

#### 2.1.2 密码登录（Legacy Auth）

**P0** 支持通过用户名（Matrix ID，格式 `@user:server.com`）和密码完成登录，对应 matrix-rust-sdk 的 `authentication::matrix` 模块。登录成功后自动创建 SQLite 加密存储（`matrix-sdk-sqlite`）。

#### 2.1.3 OAuth/OIDC 登录

**P1** 对支持 MSC3861 的服务器（如 matrix.org），使用系统浏览器（`openLink`）或内嵌 `@ohos/web` 组件完成 OAuth 授权码流程，并通过 Deep Link 回调处理 `AuthorizationCode`，对应 `authentication::oauth::OAuth::login()` API。

**P1** 支持通过 QR 码登录（`OAuth::login_with_qr_code()`），从已登录设备扫码完成新设备授权，无需输入密码。

#### 2.1.4 会话验证与加密初始化

**P0** 登录后必须完成设备验证流程：优先从已有登录设备扫码验证，或输入 Recovery Key。若无法验证，提供"重置加密"选项并明确告知后果（历史消息密钥将被清除）。

**P0** 验证完成后自动初始化 `OlmMachine`（`matrix_sdk_crypto`），启动密钥备份同步。

#### 2.1.5 退出登录

**P0** 支持退出当前设备会话（`Client::logout()`），退出时清除本地 SQLite 数据库和所有缓存。

### 2.2 房间列表模块（Room List）

#### 2.2.1 房间列表展示

**P0** 应用启动后，通过 `RoomListService` 加载并展示所有已加入的房间，按最新消息时间倒序排列。列表使用 `LazyForEach` + `@Reusable` 实现懒加载，保证在房间数量超过 500 个时仍能流畅滑动（目标帧率 60fps）。

每个房间条目展示：
- 房间头像（支持字母占位符 Avatar）
- 房间名称
- 最新消息预览（文本截断，媒体消息显示类型图标）
- 最新消息时间戳
- 未读消息数角标（区分普通未读和 @提及）
- 加密状态指示（如有 E2EE 问题显示盾牌图标）

**P0** 支持下拉刷新，触发 `RoomListService` 的增量同步。

#### 2.2.2 房间搜索与过滤

**P1** 房间列表顶部提供搜索框，支持按房间名称模糊过滤（使用 `RoomList::entries_with_dynamic_adapters()` 的动态过滤器）。

**P1** 支持按"未读"快速筛选，仅显示有未读消息的房间。

#### 2.2.3 房间操作（长按菜单）

**P1** 长按房间条目弹出操作菜单，包含：
- 标记已读 / 标记未读（`Room::set_unread_flag()`）
- 静音通知 / 取消静音
- 离开房间（`Room::leave()`）

#### 2.2.4 直接消息（DM）

**P1** 支持通过用户 Matrix ID 发起新的直接消息（`Client::get_dm_room()` 或创建新 DM 房间）。

---

### 2.3 消息时间线模块（Timeline）

#### 2.3.1 时间线渲染

**P0** 进入房间后，通过 `matrix_sdk_ui::timeline::Timeline` 加载并渲染消息时间线。时间线使用 `LazyForEach` 实现虚拟化渲染，确保在消息数量超过 5000 条时首屏加载时间不超过 800ms。

利用 API 22 新增的滚动组件**可见区域变化事件回调**（`onVisibilityChange`）精确追踪消息条目的可见状态，用于发送已读回执（`m.read`），替代原有基于滚动位置估算的逻辑。

时间线中包含以下虚拟条目（Virtual Items）：
- 日期分割线（按天分组）
- 未读消息分割线（标记上次阅读位置）
- 加载中指示器（分页加载时）

#### 2.3.2 消息类型支持

所有消息类型对应 `TimelineItemContent` 枚举的各变体：

| 消息类型                       | 优先级 | 渲染说明                                            |
| :----------------------------- | :----: | :-------------------------------------------------- |
| 文本消息（`m.text`）           | **P0** | 支持 Markdown 基础渲染（粗体、斜体、代码块、链接）  |
| 图片消息（`m.image`）          | **P0** | 缩略图展示，点击全屏预览，支持 E2EE 解密展示        |
| 视频消息（`m.video`）          | **P1** | 缩略图 + 时长展示，点击通过 AVPlayer 播放           |
| 音频消息（`m.audio`）          | **P1** | 波形图展示（`BaseAudioInfo::waveform`），内嵌播放器 |
| 语音消息（`m.audio` + voice）  | **P1** | 同音频，额外展示录音时长                            |
| 文件消息（`m.file`）           | **P1** | 文件名 + 大小展示，点击下载并调用系统打开           |
| 贴纸（`m.sticker`）            | **P2** | 图片渲染，无操作栏                                  |
| 位置消息（`m.location`）       | **P2** | 展示静态地图截图，点击跳转地图应用                  |
| 加密失败（`m.room.encrypted`） | **P0** | 展示"无法解密此消息"提示，附带原因说明              |
| 房间状态事件                   | **P1** | 折叠展示（如"XXX 加入了房间"）                      |
| 未知消息类型                   | **P0** | 展示降级占位符（"不支持此消息类型"）                |

#### 2.3.3 消息气泡 UI 规范

**P0** 消息气泡严格遵循 Compound 设计系统规范（参考 Figma 文件 ElementX Design System v1.0）：
- 自己发送的消息显示在右侧，他人消息在左侧
- 消息状态指示：发送中（单勾）、已发送（双勾）、发送失败（红色感叹号）
- 消息时间戳显示在气泡右下角

**P0** 消息气泡支持长按弹出操作菜单（见 2.3.6 节）。

#### 2.3.4 消息回复（Reply）

**P0** 支持向上滑动消息或长按选择"回复"，在输入框上方展示被引用消息预览。发送时通过 `Room::send()` 携带 `in_reply_to` 关联。

**P0** 时间线中正确渲染回复引用气泡，点击可跳转到原始消息。

#### 2.3.5 消息反应（Reactions）

**P1** 支持长按消息弹出 Emoji 选择器，发送 `m.reaction` 事件。

**P1** 消息气泡底部展示聚合的 Emoji 反应（`ReactionsByKeyBySender`），点击可切换自己的反应状态。

#### 2.3.6 消息操作菜单

**P1** 长按消息气泡弹出操作菜单，包含：
- 回复
- 编辑（仅自己发送的消息）
- 删除（发送 `m.room.redaction`）
- 复制文本
- 转发
- 查看详情（消息 ID、加密信息等）

#### 2.3.7 消息编辑

**P1** 编辑自己发送的消息，输入框预填原始内容，发送后时间线展示"（已编辑）"标记。

#### 2.3.8 消息分页加载

**P0** 时间线支持向上滚动时自动触发历史消息分页加载（`RoomPagination::run_backwards_once()`），加载完成后平滑插入消息，不打断当前阅读位置。

#### 2.3.9 跳转与定位

**P1** 支持点击回复引用跳转到原始消息位置，若消息不在当前加载范围内则通过 `TimelineFocus::Event` 重新加载该消息上下文。

### 2.4 消息输入模块（Composer）

#### 2.4.1 文本输入

**P0** 提供富文本输入框（基于 `RichEditor` 组件，使用 API 22 新增的 `maxLength` 行数控制 API），支持：
- 多行输入，自动扩展高度（最大 4 行）
- 发送按钮（点击或回车发送）
- Markdown 快捷格式（`**粗体**`、`_斜体_`、`` `代码` ``）
- 利用 API 22 滚动内容区域偏移量接口实现输入框边缘留白效果

#### 2.4.2 媒体发送

**P1** 点击附件图标弹出选择菜单：
- 从相册选择图片/视频（通过 `PhotoAccessHelper` FilePicker）
- 拍摄照片/视频（调用系统相机）
- 选择文件（通过系统 FilePicker，返回沙箱 URI 后读取内容上传）

所有媒体文件在上传前通过 `Room::send_attachment()` 进行 E2EE 加密。

#### 2.4.3 语音录制

**P2** 长按麦克风按钮录制语音消息，松开发送，上滑取消。录制时展示实时波形动画。

#### 2.4.4 输入状态

**P1** 用户输入时向房间发送 `m.typing` 事件，时间线顶部展示"XXX 正在输入..."提示，3 秒无输入后自动停止发送。

#### 2.4.5 草稿保存

**P2** 切换房间时自动保存当前输入框的草稿内容，返回时恢复。

### 2.5 端到端加密模块（E2EE）

#### 2.5.1 加密消息发送与接收

**P0** 在加密房间中，所有消息通过 `vodozemac`（Megolm 协议）自动加密发送，接收时自动解密展示，用户无感知（Invisible Encryption）。

#### 2.5.2 密钥备份

**P0** 支持 4S（Secret Storage and Sharing）密钥备份，登录后自动从服务器拉取密钥备份。

**P0** 支持设置 Recovery Key，引导用户导出并妥善保存。

#### 2.5.3 设备验证

**P1** 支持通过 Emoji 表情对比（SAS）验证其他设备。

**P1** 对未验证设备发送的消息展示安全盾牌警告（`ShieldState`）。

#### 2.5.4 解密失败处理

**P0** 对无法解密的消息展示友好提示，根据 `UtdCause` 枚举给出具体原因，提供"请求重发密钥"按钮。

### 2.6 通知模块（Notifications）

#### 2.6.1 本地通知

**P0** 应用在前台时，通过 `Notification Kit` 展示新消息的横幅通知，包含房间名称、发送者和消息预览。

#### 2.6.2 远程推送（Push Kit）【优先级已升级为 P1】

**P1** 接入华为 Push Kit，通过 AppGallery Connect 注册推送 token，并将 token 注册到 Matrix 服务器的 Pusher 接口（`/_matrix/client/v3/pushers/set`）。应用在后台或关闭时，通过华为推送服务接收新消息通知，点击通知直接跳转到对应房间。

利用 API 22 `Notification Kit` 新增的三方应用获取本机通知能力，支持通知自动同步至用户的华为手表等穿戴设备，无需额外开发。

#### 2.6.3 通知规则

**P1** 遵循 Matrix Push Rules 规范，支持以下默认规则：
- @提及消息：高优先级通知（带声音）
- 关键词匹配：高优先级通知
- DM 消息：高优先级通知
- 普通群消息：静默通知（仅角标）

**P1** 支持对单个房间设置通知级别（全部消息 / 仅提及 / 静音）。

## 第三章 非功能需求

### 3.1 性能指标

| 指标             | 目标值  | 测试场景                     |
| :--------------- | :------ | :--------------------------- |
| 冷启动时间       | ≤ 2 秒  | 从图标点击到房间列表可交互   |
| 房间列表滑动帧率 | ≥ 60fps | 500 个房间，LazyForEach 渲染 |
| 时间线首屏加载   | ≤ 800ms | 5000 条历史消息的房间        |
| 消息发送延迟     | ≤ 200ms | 本地回显（Local Echo）时间   |
| 内存占用         | ≤ 300MB | 正常使用状态，10 个活跃房间  |

### 3.2 安全要求

- 所有端到端加密逻辑必须在 Rust 层（`matrix_sdk_crypto` + `vodozemac`）完成，ArkTS 层不得接触任何明文密钥材料。
- SQLite 数据库必须启用加密（`StoreCipher`），密钥存储于 `Asset Store Kit`（鸿蒙安全存储）。
- 所有网络通信强制使用 TLS 1.2+（`rustls`），禁止 HTTP 明文连接。
- 所有 SQLite 操作统一路由到单一 tokio 线程，规避 SQLite 锁竞争问题（SDK Issue #5160）。
- 利用 API 22 新增 Node-API 强引用接口（`napi_create_strong_reference`）管理 NAPI 回调生命周期，消除跨语言崩溃风险。

### 3.3 兼容性要求

- **最低 API 版本**：HarmonyOS API 22（HarmonyOS 6.0.2）
- **目标 API 版本**：HarmonyOS API 22（HarmonyOS 6.0.2）
- **Matrix 协议版本**：支持 Matrix Spec v1.10+，要求服务器支持 MSC4186（Native Sliding Sync）
- **服务器兼容性**：支持 Synapse 1.100+、Conduit 0.8+、Dendrite（有限支持）

### 3.4 可访问性

**P2** 支持鸿蒙无障碍服务，消息气泡、按钮等交互元素提供 `accessibilityText` 描述，支持屏幕朗读。

### 3.5 调试与可观测性

利用 API 22 `Performance Analysis Kit` 新增能力：
- 配置 `HiAppEvent` 主线程超时事件策略，自动采样卡顿栈
- 启用 JS Crash `HybridStack` 混合栈，支持打印 Rust（CPP）→ ArkTS（JS）跨语言调用栈，大幅降低跨层 Bug 定位成本
- 启用 AppFreeze libuv 异步栈跟踪，监控 tokio 异步任务的阻塞情况

## 第四章 技术架构约束

### 4.1 分层架构

```
┌─────────────────────────────────┐
│        ArkUI（UI 层）            │  ArkTS + Compound 设计系统
│   LazyForEach / @Reusable       │
├─────────────────────────────────┤
│      ArkTS 状态管理层            │  Worker 线程 + TaskPool
│   AppStorage / @State / @Link   │
├─────────────────────────────────┤
│   ohos-rs NAPI Bridge（桥接层）  │  Rust → ArkTS 类型映射
│   ThreadSafeFunction / Promise  │  API 22 强引用生命周期管理
├─────────────────────────────────┤
│   matrix-rust-sdk 0.16.0（核心）│  tokio async runtime
│   RoomListService / Timeline    │
│   matrix_sdk_crypto / SQLite    │
└─────────────────────────────────┘
```

### 4.2 关键技术决策

- **TLS 方案**：强制使用 `rustls` + `rustls-platform-verifier`，禁止 `native-tls`，规避 OpenSSL 交叉编译风险。
- **异步运行时**：tokio 运行时在独立 Worker 线程中初始化，通过 `ThreadSafeFunction` 向主线程推送事件。
- **数据序列化**：Rust-ArkTS 跨层数据传递使用 JSON 序列化（`serde_json`），复杂对象映射为 ArkTS `interface`。
- **SQLite 并发**：所有数据库操作通过单一 Mutex 保护的 tokio 任务执行，禁止并发写入。
- **NAPI 生命周期**：使用 API 22 新增 Node-API 强引用接口管理 ArkTS 回调对象，防止 GC 提前回收。
- **Cargo.lock 锁定**：严格锁定 matrix-rust-sdk 0.16.0，禁止宽松版本约束。

| 阶段            | 里程碑         | 核心交付物                                                | 验收标准                                            |
| :-------------- | :------------- | :-------------------------------------------------------- | :-------------------------------------------------- |
| **P0 原型验证** | 跨语言链路闭环 | ohos-rs Hello World，含 `ring` 依赖的 Rust 函数成功调用   | 在鸿蒙模拟器上运行，无 panic                        |
